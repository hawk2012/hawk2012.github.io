В последние годы контейнеризация стала неотъемлемой частью разработки программного обеспечения. Одной из самых популярных платформ для работы с контейнерами является Docker. Однако, существуют альтернативы, такие как **Podman** и **Buildah**, которые предлагают opensource-решения для управления контейнерами без использования Docker Engine. В этой статье мы рассмотрим, как использовать эти инструменты для создания и запуска микросервиса на базе .NET Core с использованием открытых портов MySQL, SSL и веб-сервера, а также создадим файл `.service` для автоматического запуска сервиса при помощи Podman.

### Что такое Podman?

**Podman** – это инструмент для управления контейнерами, который позволяет создавать, запускать и управлять контейнерами и образами без необходимости использования демона Docker. Он поддерживает большинство команд Docker, что делает переход от одного инструмента к другому достаточно простым. Одним из ключевых преимуществ Podman перед Docker является то, что он работает в режиме rootless, что повышает безопасность системы за счет ограничения привилегий процесса.

### Что такое Buildah?

**Buildah** – это утилита для сборки образов контейнеров. Она предоставляет низкоуровневый API для создания образов, что позволяет разработчикам создавать образы контейнеров без необходимости запуска промежуточного контейнера. Это упрощает процесс сборки и уменьшает количество шагов, необходимых для создания образа.

### Установка Podman и Buildah

Для начала необходимо установить Podman и Buildah. Эти компоненты уже содержатся во всех последних Linux, например Debian 11. Рассмотрим установку на примере операционной системы Ubuntu:

```bash
sudo apt update
sudo apt install podman buildah
```

После установки вы можете проверить версии установленных инструментов:

```bash
podman --version
buildah --version
```

### Создание образа контейнера с микросервисом .NET Core

Для того чтобы создать образ контейнера с микросервисом на основе .NET Core, нам потребуется Dockerfile. Давайте создадим простой пример Dockerfile для нашего приложения:

```dockerfile
FROM mcr.microsoft.com/dotnet/core/sdk:3.1 AS build
WORKDIR /app
COPY *.csproj .
RUN dotnet restore
COPY . .
RUN dotnet publish -c Release -o out

FROM mcr.microsoft.com/dotnet/core/aspnet:3.1
WORKDIR /app
COPY --from=build /app/out .
EXPOSE 80
CMD ["dotnet", "MyApp.dll"]
```

Этот Dockerfile использует многоступенчатую сборку: сначала мы собираем проект, затем копируем собранные файлы в финальный образ. Теперь мы можем собрать этот образ с помощью команды `buildah`:

```bash
buildah bud -f Dockerfile -t myapp:latest .
```

Эта команда создаст образ контейнера под названием `myapp:latest`.

### Запуск контейнера с открытыми портами

Теперь, когда у нас есть собранный образ, мы можем запустить контейнер с открытыми портами для MySQL, SSL и веб-сервера. Для этого воспользуемся командой `podman run`:

```bash
podman run -d \
  -p 3306:3306 \ # Порт MySQL
  -p 443:443 \   # Порт SSL
  -p 80:80 \     # Порт HTTP
  --name myapp \
  myapp:latest
```

Эта команда запустит контейнер в фоновом режиме (`-d`), откроет указанные порты и присвоит ему имя `myapp`. Вы можете проверить статус запущенного контейнера с помощью команды:

```bash
podman ps
```

### Настройка файла `.service` для автоматического запуска микросервиса

Чтобы обеспечить автоматический запуск нашего микросервиса при загрузке системы, мы можем создать системную службу с использованием systemd. Создадим файл `/etc/systemd/system/myapp.service` со следующим содержимым:

```ini
[Unit]
Description=MyApp Service
After=network.target

[Service]
Type=simple
ExecStart=/usr/bin/podman start myapp
ExecStop=/usr/bin/podman stop myapp
Restart=always

[Install]
WantedBy=multi-user.target
```

Затем активируем и включим эту службу:

```bash
systemctl enable myapp.service
systemctl start myapp.service
```

Теперь наш микросервис будет автоматически запускаться при каждой загрузке системы.
